# DICOM StoreSCP 现场联调工具项目开发计划

## 1. 文档目的

本文档用于明确 DICOM StoreSCP 现场联调工具项目的开发阶段、任务拆解、里程碑、交付物和推进顺序，作为项目执行与跟踪依据。

## 2. 项目开发目标

围绕现场联调需求，逐步完成一款可在 Windows 笔记本运行的 DICOM 调试工具，具备：

- StoreSCP 接收能力
- 完整日志能力
- 基础网络调试能力
- 可运行的桌面应用形态

## 3. 开发原则

- 先完成最小可用版本，再逐步增强
- 优先实现现场最常用功能
- 每阶段均形成可验证结果
- 文档、设计、开发、验证同步推进

## 4. 阶段划分

项目建议分为 5 个阶段实施。

### 4.1 阶段一：需求与方案确认

目标：

- 完成项目范围收敛
- 明确技术路线与实现边界

主要任务：

- 完成立项文档
- 完成项目实现方案
- 完成项目开发计划
- 完成需求规格说明书
- 完成软件架构设计说明书

交付物：

- 项目立项书
- 项目实现方案
- 项目开发计划
- 软件需求规格说明书
- 软件架构设计说明书

### 4.2 阶段二：开发环境与项目初始化

目标：

- 搭建可开发、可运行、可构建的工程基础

主要任务：

- 安装 .NET SDK
- 安装 Visual Studio 开发环境
- 初始化 Git 仓库
- 创建 WPF 项目骨架
- 引入 fo-dicom、Serilog 等依赖
- 建立基础目录结构
- 建立配置与日志基础设施

交付物：

- 可编译运行的项目骨架
- 初始代码仓库

### 4.3 阶段三：MVP 功能开发

目标：

- 形成可现场使用的最小可用版本

主要任务：

- 实现主界面
- 实现监听启动与停止
- 实现 StoreSCP 图像接收与文件落盘
- 实现实时日志显示
- 实现 Ping 功能
- 实现端口连通性检测
- 实现 C-ECHO 功能
- 实现基础配置保存与加载
- 完成基础自测

交付物：

- MVP 可执行程序
- 基础测试记录

### 4.4 阶段四：增强功能开发

目标：

- 提升可用性、可追溯性和问题定位能力

主要任务：

- 增加接收会话历史记录
- 引入 SQLite 存储
- 增加日志查询与筛选
- 增加日志导出
- 优化异常提示与错误分类
- 增加接收详情展示

交付物：

- 增强版可执行程序
- 增强版测试记录

### 4.5 阶段五：完善与交付

目标：

- 完成产品化包装和交付准备

主要任务：

- 增加测试报告导出
- 完善用户使用说明
- 制作安装包或自包含发布包
- 完成回归测试
- 形成发布版本

交付物：

- 正式发布包
- 使用说明
- 发布记录

## 6. 里程碑建议

建议设置以下里程碑：

### 里程碑 M1：方案冻结

完成标志：

- 立项书、实现方案、开发计划、SRS、架构设计完成

### 里程碑 M2：项目骨架完成

完成标志：

- 本地开发环境可用
- 项目可编译启动
- 基础界面显示正常

### 里程碑 M3：MVP 可运行

完成标志：

- StoreSCP 可接收图像
- 日志可查看
- Ping、端口检测、C-ECHO 可用

### 里程碑 M4：增强版完成

完成标志：

- 接收记录、日志导出、异常提示优化完成

### 里程碑 M5：发布版完成

完成标志：

- 安装包或发布包生成
- 测试通过
- 形成交付版本

## 8. 风险与应对

### 8.1 技术风险

- 可能存在 DICOM 库与目标框架兼容问题
- 不同 CT 厂家兼容性存在差异

应对：

- 优先选择成熟稳定的库版本
- 尽早做真实设备联调验证

### 8.2 环境风险

- 当前电脑缺少开发环境
- 医院现场网络条件复杂

应对：

- 先完成环境搭建清单
- 将网络诊断作为 MVP 核心能力

### 8.3 需求风险

- 后期可能新增很多附加需求

应对：

- 明确 Phase 1 范围
- 非核心需求进入后续阶段

## 9. 当前建议执行顺序

基于当前项目状态，建议按以下顺序执行：

1. 补齐项目文档
2. 编写需求规格说明书
3. 编写软件架构设计说明书
4. 安装开发环境
5. 初始化项目代码
6. 开发 MVP
7. 做现场模拟验证
8. 增强功能与打包发布

## 10. 结论

本项目应按文档先行、MVP 优先、分阶段交付的方式推进。当前最合理的下一步是继续补齐需求与架构文档，随后再进入环境搭建和代码实现阶段，以保证项目实施过程清晰、稳定、可追溯。
