# DICOM StoreSCP 现场联调工具项目立项书

## 1. 项目名称

DICOM StoreSCP 现场联调工具

## 2. 项目背景

当前产品具备 StoreSCP 功能，可接收 CT 端推送的 DICOM 图像。现场调试时，通常需要将整机搬运至 CT 机房，通过 CT 向设备推送图像来验证网络通信与接收功能。该方式存在以下问题：

- 设备体积较大，搬运成本高
- 现场联调效率低，重复工作多
- 问题定位依赖整机环境，不利于快速排障
- 缺少便携、标准化的联调工具，现场取证能力不足

为提升现场调试效率，拟开发一款可运行于 Windows 笔记本的 DICOM StoreSCP 现场联调工具，用于替代整机完成 CT 推送联调、网络验证和日志取证。

## 3. 项目目标

开发一款 Windows 应用程序，支持 DICOM StoreSCP 接收、C-ECHO 测试及基础网络诊断，具备完整日志记录和测试结果留存能力，满足医院现场快速联调与故障定位需求。

## 4. 项目定位

本工具定位为医疗现场安装、调试、维护辅助软件，非诊断用途。

主要用于：

- 设备安装调试
- CT 联网验证
- DICOM 通信排障
- 现场日志采集与问题取证

不用于：

- 医学影像诊断
- 正式影像归档
- 临床业务处理

## 5. 建设必要性

开发该工具具有以下必要性：

- 降低现场调试对整机设备的依赖
- 提升 CT 联调效率，减少人员和时间成本
- 通过标准化日志和测试记录提升问题复现与定位效率
- 为售后、实施、研发提供统一的现场验证手段
- 为后续产品验收、内部测试、质量追溯提供辅助支撑

## 6. 使用场景

典型使用场景包括：

- 医院现场验证 CT 是否能向目标 AE 正常推送图像
- 验证本地网络、IP、端口、防火墙、AE Title 配置是否正确
- 在设备未到场或不便搬运时，使用笔记本临时替代接收端
- 在故障排查时，通过日志分析关联建立、图像发送、异常中断等问题
- 对现场测试过程进行留痕，形成联调记录

## 7. 项目范围

### 7.1 纳入范围

- Windows 桌面应用程序
- DICOM StoreSCP 接收功能
- DICOM C-ECHO 测试功能
- 基础网络诊断功能，如 Ping、端口连通性检测
- 日志记录、查询与导出
- 接收文件保存与接收结果展示

### 7.2 暂不纳入范围

- PACS 归档管理
- 诊断级图像浏览
- HIS/RIS 对接
- 云端同步
- 高级 DICOM 工作流功能，如完整的 Modality Worklist、Storage Commitment

## 8. 功能需求

### 8.1 StoreSCP 接收功能

系统应支持：

- 配置本地 AE Title
- 配置监听端口
- 启动与停止 DICOM 监听服务
- 接收 CT 推送的 DICOM 图像
- 保存接收到的图像文件至本地目录
- 展示本次接收的图像数量、接收时间、来源 AE、来源 IP 等信息
- 支持常见 CT 影像存储类 SOP

### 8.2 日志功能

系统应支持：

- 记录应用启动、停止、配置变更等操作日志
- 记录 DICOM 关联建立、释放、拒绝、异常等协议日志
- 记录图像接收成功、失败、文件保存异常等业务日志
- 支持日志按时间查看、筛选和导出
- 支持在现场导出联调日志用于问题反馈

### 8.3 网络调试功能

系统应支持：

- 对目标 IP 执行 Ping
- 对目标 DICOM 节点执行 C-ECHO
- 检测指定 IP 和端口是否连通
- 展示本机网卡和 IP 信息，辅助选择监听地址

### 8.4 结果展示功能

系统应支持：

- 展示当前监听状态
- 展示最近接收会话记录
- 展示接收文件基本信息
- 展示测试成功或失败结果及错误提示

## 9. 非功能需求

### 9.1 易用性

- 软件应提供图形界面
- 现场工程师无需命令行即可完成常用操作
- 界面布局应清晰，适合快速联调

### 9.2 可维护性

- 软件架构清晰，便于后续扩展 StoreSCU、DICOM 标签查看等功能
- 日志结构规范，便于研发定位问题

### 9.3 可靠性

- 软件异常退出时应尽可能保留日志
- 对网络中断、端口占用、AE 配置错误等场景应给出明确提示

### 9.4 兼容性

- 支持 Windows 10/11
- 可在普通笔记本环境运行
- 支持常见医院局域网环境

### 9.5 安全性

- 本地数据保存应可控
- 对涉及患者信息的数据应支持最小化展示与后续脱敏扩展能力

## 10. 总体方案

建议采用 Windows 桌面应用架构，软件由以下模块组成：

- UI 界面模块
- DICOM 通信模块
- 网络诊断模块
- 日志管理模块
- 文件存储模块
- 配置管理模块

建议技术路线如下：

- 开发语言：C#
- 桌面框架：WPF 或 WinUI 3
- DICOM 库：fo-dicom
- 日志组件：Serilog
- 数据存储：SQLite 或本地文件

## 11. 界面规划

建议包含以下页面：

### 11.1 监听页

- 配置 AE Title、端口、本机 IP
- 启动或停止监听
- 显示运行状态

### 11.2 网络测试页

- Ping
- 端口检测
- C-ECHO

### 11.3 接收记录页

- 展示接收会话、来源、数量、状态

### 11.4 日志页

- 实时日志
- 筛选查询
- 导出日志

## 12. 预期效益

项目上线后，预期达到以下效果：

- 现场联调无需搬运整机，显著降低实施成本
- 缩短 CT 联调和问题定位时间
- 统一工程师现场操作流程
- 提升现场问题记录与复盘能力
- 为售后支持与质量追溯提供工具支撑

## 13. 项目实施建议

建议按三阶段推进：

### 13.1 阶段一：MVP

- 完成 StoreSCP、日志、Ping、C-ECHO、GUI 基础界面
- 满足最核心现场测试需求

### 13.2 阶段二：增强

- 增加接收记录管理、日志导出、错误提示优化、SQLite 历史记录

### 13.3 阶段三：完善

- 增加测试报告导出、标签查看、脱敏能力、安装包发布

## 14. 风险与关注点

- 现场网络环境复杂，多网卡、VLAN、防火墙可能影响测试效果
- 不同 CT 厂家对 AE Title、传输语法、连接策略存在差异
- 如工具纳入质量体系，需明确其用途边界和文档要求
- 若涉及真实患者数据，需注意信息保护与日志脱敏

## 15. 立项结论

本项目具有明确的现场应用价值和较高的实施可行性。通过开发便携式 Windows 版 DICOM StoreSCP 联调工具，可有效替代整机参与 CT 联调测试，显著提升现场实施效率和问题定位能力，建议立项实施。

## 16. 一句话目标

开发一款可在 Windows 笔记本上运行的 DICOM 调试工具，支持 StoreSCP 接收、C-ECHO 和基础网络诊断，并提供完整日志与测试报告能力，用于替代现场搬运设备进行 CT 联调测试。
