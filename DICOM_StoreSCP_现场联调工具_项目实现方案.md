# DICOM StoreSCP 现场联调工具项目实现方案

## 1. 文档目的

本文档用于明确 DICOM StoreSCP 现场联调工具的技术实现思路、系统架构、模块划分、关键流程和实施策略，作为项目开发与落地执行的依据。

## 2. 项目目标

开发一款可在 Windows 笔记本上运行的桌面应用，支持以下能力：

- 提供 DICOM StoreSCP 接收功能
- 提供完善的日志记录与问题追溯能力
- 提供基础网络调试能力，包括 Ping、端口检测、C-ECHO
- 形成可安装、可运行、可维护的 Windows 应用程序

## 3. 总体实现思路

本项目采用 Windows 桌面应用架构，以图形界面为入口，以 DICOM 通信服务为核心，以日志与网络诊断为辅助，形成适用于医院现场的便携式联调工具。

实现上遵循以下原则：

- 先实现最小可用版本，再逐步增强
- 通信能力、界面能力、日志能力分层解耦
- 优先保证现场稳定性、易用性和问题定位能力
- 优先采用成熟技术栈，降低开发与维护风险

## 4. 技术选型方案

### 4.1 开发语言

C#

选择原因：

- 与 Windows 桌面开发适配度高
- 网络、线程、日志、文件处理能力成熟
- 与 DICOM 相关开源库集成方便

### 4.2 UI 框架

WPF

选择原因：

- Windows 桌面应用成熟稳定
- 数据绑定与界面开发效率较高
- 适合工具类软件快速构建
- 社区资料丰富，维护成本较低

说明：

WinUI 3 也可选，但考虑稳定性、开发门槛和快速落地，优先采用 WPF。

### 4.3 DICOM 通信库

fo-dicom

选择原因：

- 支持 DICOM SCU/SCP 功能
- 支持 C-ECHO、C-STORE 等常用能力
- 在 .NET 生态中成熟度较高

### 4.4 日志组件

Serilog

选择原因：

- 结构化日志支持较好
- 易于写入文件、控制台和界面
- 便于后续增加日志导出与检索能力

### 4.5 数据存储

分阶段采用如下方式：

- MVP 阶段：本地文件 + JSON 配置
- 增强阶段：SQLite 存储接收会话与历史记录

### 4.6 打包方式

优先方案：

- 自包含发布包

后续可选：

- MSI 或 MSIX 安装包

## 5. 系统架构设计

建议采用分层架构。

### 5.1 架构分层

- 表现层：WPF 图形界面
- 应用层：业务编排与用例服务
- 领域能力层：DICOM 接收、网络测试、日志、配置、文件管理
- 基础设施层：fo-dicom、Serilog、SQLite、文件系统、系统网络接口

### 5.2 模块划分

建议划分以下核心模块：

- UI 模块
- DICOM 接收模块
- DICOM 网络测试模块
- 基础网络诊断模块
- 日志管理模块
- 配置管理模块
- 文件存储模块
- 报告导出模块

## 6. 功能模块实现方案

### 6.1 UI 模块

负责：

- 展示系统状态
- 接收用户配置输入
- 提供监听启动与停止操作
- 展示接收结果、测试结果和日志

建议页面：

- 首页/监听页
- 网络测试页
- 接收记录页
- 日志页
- 设置页

### 6.2 DICOM 接收模块

负责：

- 创建并启动 StoreSCP 服务
- 监听指定端口
- 处理 DICOM Association 建立与释放
- 接收 C-STORE 请求
- 将文件保存到本地目录
- 返回接收结果并写入日志

建议能力点：

- 支持配置 AE Title
- 支持监听端口配置
- 支持 Called AE 校验
- 支持 Calling AE 白名单扩展
- 支持按照日期/患者/检查分目录保存

### 6.3 DICOM 网络测试模块

负责：

- 发起 C-ECHO 测试
- 验证目标 AE 是否可建立 DICOM 连接
- 记录连接耗时与失败原因

建议输入项：

- 目标 IP
- 目标端口
- 目标 AE Title
- 本地 AE Title

### 6.4 基础网络诊断模块

负责：

- Ping 目标 IP
- 检测目标端口是否连通
- 获取本机网卡和 IP 信息

建议输出：

- 网络通断结果
- 延迟信息
- 端口探测结果
- 本机候选 IP 列表

### 6.5 日志管理模块

负责：

- 记录应用运行日志
- 记录 DICOM 协议日志
- 记录异常与错误堆栈
- 提供日志查询与导出

日志建议分类：

- 操作日志
- 协议日志
- 错误日志

### 6.6 配置管理模块

负责：

- 读取和保存本地配置
- 管理默认参数
- 启动时加载上次配置

建议配置项：

- 本地 AE Title
- 监听端口
- 接收目录
- 日志目录
- 是否校验 Called AE
- 是否开启详细协议日志

### 6.7 文件存储模块

负责：

- 组织接收文件目录
- 确保目录存在
- 保存 DICOM 文件
- 返回保存路径与状态

建议目录结构：

```text
AppData/
  config/
  logs/
  received/
    2026-04-03/
      STUDY_xxx/
  export/
```

### 6.8 报告导出模块

增强阶段负责：

- 汇总测试结果
- 导出日志与接收摘要
- 生成现场联调报告

## 7. 关键流程设计

### 7.1 StoreSCP 启动流程

1. 用户输入本地 AE Title、监听端口和接收目录
2. 系统校验参数合法性
3. 系统绑定监听端口
4. 创建 DICOM SCP 服务
5. 启动监听并更新界面状态
6. 写入启动日志

### 7.2 图像接收流程

1. CT 发起 Association
2. 系统校验 Called AE 与连接参数
3. 系统接受 C-STORE 请求
4. 解析实例基础信息
5. 生成保存路径
6. 落盘保存 DICOM 文件
7. 记录实例接收日志
8. 会话结束后汇总显示接收结果

### 7.3 C-ECHO 测试流程

1. 用户输入目标节点参数
2. 系统构建 DICOM Echo Client
3. 发起连接与 C-ECHO 请求
4. 返回成功或失败状态
5. 展示测试耗时与错误原因
6. 写入测试日志

### 7.4 日志导出流程

1. 用户选择导出范围
2. 系统收集日志和测试记录
3. 生成导出目录或压缩包
4. 返回导出结果

## 8. 版本实施策略

### 8.1 Phase 1：MVP

实现范围：

- WPF 主界面骨架
- StoreSCP 启动与停止
- DICOM 文件接收与落盘
- 实时日志显示
- Ping
- 端口检测
- C-ECHO
- 基础配置保存

目标：

形成可现场使用的最小可用版本。

### 8.2 Phase 2：增强版

实现范围：

- 接收会话历史记录
- SQLite 存储
- 更细粒度异常提示
- 日志导出
- 接收详情查看

目标：

提升问题追溯与使用便利性。

### 8.3 Phase 3：完善版

实现范围：

- 测试报告导出
- DICOM 标签浏览
- 基础图像预览
- 脱敏能力
- 安装包制作

目标：

提升产品化程度和交付完整性。

## 9. 异常场景与处理策略

重点处理以下问题：

- 端口被占用
- 本机 IP 选择错误
- 网络不可达
- 防火墙阻断
- AE Title 不匹配
- C-ECHO 失败
- Association 被拒绝
- 文件保存失败
- 接收过程中网络中断

处理策略：

- 界面给出明确错误提示
- 日志记录详细上下文
- 失败信息中尽量给出排查建议

## 10. 可测试性设计

为保证后续验证效率，设计阶段应考虑：

- 服务模块可独立测试
- 日志内容具备可追溯字段
- 配置、文件路径、网络参数可注入
- 通信模块与界面层解耦，便于做集成测试

- 操作系统：Windows 10/11

- SDK：.NET 8 SDK

## 13. 结论

本项目建议采用 C# + WPF + fo-dicom 的技术路线，按照 MVP 到增强版再到完善版的路径逐步落地。该方案能够较好满足 Windows 现场联调工具对稳定性、便携性、日志追溯和快速交付的要求，适合作为本项目的实施依据。
