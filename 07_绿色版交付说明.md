# 绿色版交付说明

最后更新时间：2026-04-04  
适用版本：当前标准发布目录 `dist\win-x64` 与 GitHub Releases 绿色包  
文档类型：交付文档

## 1. 文档目的

本文档用于说明当前版本如何以“绿色版”方式交付和使用。

这里的“绿色版”指：

- 不依赖目标电脑预装 `.NET SDK`
- 不需要复杂安装流程
- 通过解压完整发布目录即可运行

## 2. 当前推荐交付方式

当前最推荐的交付方式不是直接发送源码仓库，而是：

- 通过 GitHub Releases 页面下载绿色版压缩包

推荐入口：

- [Releases](https://github.com/kuangliu774392519/storeSCP_tool/releases)

## 3. 当前绿色版交付物

发布包命名规则：

```text
StorescpTool_win-x64_portable_<version>.zip
```

主程序位于压缩包解压后的目录中：

```text
StorescpTool.App.exe
```

## 4. 正确交付方式

应交付整个发布目录，而不是只交付单个 exe。

建议做法：

1. 从 GitHub Releases 下载 zip
2. 解压到本地任意可写目录
3. 运行 `StorescpTool.App.exe`

## 5. 不正确交付方式

不要只单独发送：

- `StorescpTool.App.exe`

不要让现场同事直接：

- `git clone` 源码仓库后尝试运行
- 从 GitHub 源码 ZIP 中查找 exe

原因：

- 当前不是单文件发布
- 运行依赖同目录下的大量 DLL 和运行时文件
- 源码仓库默认不包含发布产物

## 6. 目标电脑要求

建议目标电脑满足：

- `64 位 Windows`
- 建议 `Windows 10` 或 `Windows 11`
- 有本地文件写入权限
- 允许监听测试端口

补充说明：

- 当前发布目标是 `win-x64`
- 不建议用于 `32 位 Windows`

## 7. 建议解压位置

建议解压到普通可写目录，例如：

```text
C:\Tools\StorescpTool\
```

或：

```text
D:\StorescpTool\
```

不建议放在权限较严格的目录，例如：

```text
C:\Program Files\
```

## 8. 首次运行说明

双击运行：

```text
StorescpTool.App.exe
```

首次运行时，程序会在应用目录下自动生成运行时目录和文件，例如：

```text
config\
  appsettings.json

data\
  logs\
  received\
  state\
  export\
```

## 9. 当前默认参数

- `本地 AE Title = RC120`
- `监听端口 = 5678`
- `监听 IP = 自动选择本机优先 IPv4`
- `C-ECHO Calling AE = RC120`
- `C-ECHO Called AE = CT_AE`

注意：

- 首次启动时会使用以上默认值
- 一旦生成 `config\appsettings.json`，后续将优先加载该配置文件

## 10. 现场使用建议

建议首次使用按以下顺序操作：

1. 启动程序
2. 检查 `AE Title`、监听 IP、监听端口
3. 点击“启动监听”
4. 先做 `Ping`
5. 再做 TCP 端口检测
6. 如有需要，做 `C-ECHO`
7. 如需自检，执行本机 `C-STORE` 自测
8. 再让 CT 或模拟器向该电脑推送图像

## 11. 常见问题

### 11.1 双击后程序打不开

可能原因：

- 文件未完整解压
- 被杀毒软件或系统策略拦截

建议检查：

- 是否解压了完整目录
- 是否从 zip 内部直接运行
- 是否被 Windows SmartScreen 拦截

### 11.2 程序能开，但监听失败

可能原因：

- 端口已被占用
- 本机防火墙阻止
- 监听 IP 不正确

建议检查：

- 更换端口
- 检查防火墙入站规则
- 重新刷新监听 IP 并选择正确网卡地址

### 11.3 CT 无法推送到该电脑

可能原因：

- IP 不通
- 端口不通
- AE Title 不匹配
- CT 目标节点配置错误

建议排查顺序：

1. `Ping`
2. TCP 端口检测
3. `C-ECHO`
4. 检查 CT 上配置的目标 AE / IP / Port

## 12. 当前结论

以后对现场同事应统一这样描述：

- 源码在 Git 仓库
- 可直接运行版本在 GitHub Releases

不要再用“拉源码仓库后直接运行”的方式做交付。
