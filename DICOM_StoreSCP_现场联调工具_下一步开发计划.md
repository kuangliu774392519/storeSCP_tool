# DICOM StoreSCP 现场联调工具下一步开发计划

## 1. 计划目的

本文档用于明确当前版本之后的下一阶段开发目标、任务拆解、优先级和验收标准，指导后续连续开发。

## 2. 下一阶段目标

下一阶段目标定义为：

`将当前可编译运行的工程骨架，推进为可现场联调使用的 MVP 版本`

## 3. 下一阶段开发重点

### 3.1 P0

- 验证并打磨 StoreSCP 接收链路
- 增加日志筛选与日志导出体验
- 提升配置输入可用性和目录选择体验

### 3.2 P1

- 增加接收记录详情查看
- 增加使用说明
- 整理发布目录与交付说明

### 3.3 P2

- 增加安装包或自包含发布包
- 增加基础使用说明

## 4. 任务拆解

### 任务一：StoreSCP 实机联调准备

目标：

- 确认监听参数配置、目录保存和接收结果展示可用于真实联调

子任务：

- 校验接收文件目录结构
- 增加接收成功后更多关键信息展示
- 处理更多 Association 异常日志

### 任务二：接收记录增强

目标：

- 提升接收结果可见性和可追溯性

子任务：

- 展示患者、Study UID、SOP UID、文件路径等字段
- 增加最近接收记录列表
- 增加接收记录详情查看

### 任务三：日志增强

目标：

- 形成真正可用于排障的日志基础能力

子任务：

- 增加日志级别筛选
- 增加日志导出入口优化
- 增加异常提示和诊断信息补充

### 任务四：配置和交互增强

目标：

- 提升现场工程师使用效率

子任务：

- 增加输入合法性校验
- 增加目录选择器
- 优化状态提示

### 任务五：发布准备

目标：

- 生成可在 Windows 电脑上直接运行的交付物

子任务：

- 整理发布输出目录
- 补充交付说明
- 形成可交付版本清单

## 5. 验收标准

下一阶段完成时，至少应满足：

- 可启动并停止 StoreSCP
- 可完成 Ping、TCP 检测、C-ECHO
- 可记录日志到本地文件
- 可在界面查看接收记录
- 可通过构建和测试验证

## 6. 推荐执行顺序

建议按以下顺序推进：

1. StoreSCP 接收链路增强
2. 日志增强与异常提示
3. 配置交互增强
4. 打包准备
5. 使用说明整理

## 7. 结论

下一阶段应集中完成“可现场使用 MVP”的最后关键环节，优先解决接收闭环、日志闭环和接收记录闭环，之后再进入持久化、导出和发布阶段。
